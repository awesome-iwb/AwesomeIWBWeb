第二部分续：API 设计

新增 API 端点 + 修改现有端点

🆕 新增 API：开发者后台 (/api/dev)

── 我的项目 ──
GET   /api/dev/projects          ← 我参与的项目列表
GET   /api/dev/projects/:id     ← 项目详情（仅自己参与的）
PATCH /api/dev/projects/:id      ← 编辑自己的项目

── 项目成员管理 ──
GET   /api/dev/projects/:id/members            ← 查看项目成员
POST  /api/dev/projects/:id/members             ← 邀请协作者
DELETE /api/dev/projects/:id/members/:userId ← 移除协作者

── Bug 反馈管理 ──
GET   /api/dev/feedback           ← 我项目的 Bug 列表
PATCH /api/dev/feedback/:id       ← 修改 Bug 状态/标签
POST  /api/dev/feedback/:id/replies    ← 回复 Bug 反馈

── 评论管理 ──
GET   /api/dev/comments           ← 我项目的评论列表
DELETE /api/dev/comments/:id       ← 删除评论
POST  /api/dev/comments/:id/replies    ← 回复评论

── 项目数据 ──
GET   /api/dev/stats              ← 我项目的统计数据
GET   /api/dev/stats/:projectId     ← 单项目详细数据

── 组织管理 ──
GET   /api/dev/organizations            ← 我的组织列表
POST  /api/dev/organizations             ← 申请创建组织
GET   /api/dev/organizations/:id       ← 组织详情
PATCH /api/dev/organizations/:id      ← 编辑组织信息（owner/admin）
GET   /api/dev/organizations/:id/members ← 组织成员列表
POST  /api/dev/organizations/:id/members ← 邀请成员（owner/admin）
DELETE /api/dev/organizations/:id/members/:userId ← 移除成员
PATCH /api/dev/organizations/:id/members/:userId ← 修改成员角色

── 项目认领 ──
POST  /api/dev/project-claims          ← 申请认领项目
GET   /api/dev/project-claims          ← 我的认领申请

🆕 新增 API：运维后台扩展 (/api/admin)

── 组织审核 ──
GET   /api/admin/organizations           ← 组织列表（含 pending）
GET   /api/admin/organizations/:id     ← 组织详情
POST  /api/admin/organizations/:id/approve ← 批准组织
POST  /api/admin/organizations/:id/reject  ← 拒绝组织
PATCH /api/admin/organizations/:id      ← 编辑组织（暂停等）
DELETE /api/admin/organizations/:id     ← 删除组织

── 项目认领审核 ──
GET   /api/admin/project-claims        ← 认领申请列表
POST  /api/admin/project-claims/:id/approve ← 批准认领
POST  /api/admin/project-claims/:id/reject  ← 拒绝认领

── 项目成员管理（运维指定开发者）──
POST  /api/admin/projects/:id/members    ← 指定项目开发者
DELETE /api/admin/projects/:id/members/:memberId ← 移除项目开发者

🔄 修改现有 API

POST /api/submissions — 提交项目后自动创建 project_members 记录（submitter 为 owner），自动授予 dev_panel_access
POST /api/dev/submissions — 开发者提交项目更新后，校验是否为项目成员
POST /api/feedback — 提交反馈前校验 user:feedback 能力
POST /api/upload — 上传图片前校验 user:avatar 能力（头像场景）
GET /api/auth/me — 返回结果增加 organizations 字段（用户所属组织列表）
GET /api/projects/:name — 返回结果增加 developers 字段（项目开发者列表，含组织）
GET /api/admin/projects/:id — 返回结果增加 members 字段，支持在项目编辑页指定开发者
PATCH /api/admin/users/:id/role — 当用户获得项目成员身份时，自动将 role 从 user 提升为 dev

🔐 权限校验逻辑变更

现有逻辑：
requireCapability('project:update') → 全局能力，有就能改所有项目
新逻辑：
requireCapability('dev:project_edit') → 全局能力 + 资源级校验
即：先检查是否有 dev:project_edit 能力，再检查是否为该项目的成员
新增辅助函数：requireProjectMember(projectIdParam)
自动从路由参数取 projectId，查 project_members 表验证

📌 自动提权逻辑

触发条件：用户首次成为项目成员（owner 或 collaborator）
执行操作：
1. 将 users.role 从 'user' 更新为 'dev'
2. 授予 dev_panel_access + dev:* 系列能力
3. 创建通知告知用户

降权逻辑：当用户不再是任何项目的成员时
1. 将 users.role 从 'dev' 更新为 'user'
2. 撤销 dev_panel_access + dev:* 系列能力
3. 创建通知告知用户

⚠️ 注意：运维手动分配的 dev 权限不受此逻辑影响（通过 user_capabilities 表保留）
