🖥️ 开发者后台导航结构

Dev 后台
📋 总览

🚪 退出
📋 总览 / Dashboard
• 我参与的组织数量 + 项目数量
• 待处理的 Bug 数量 + 待回复的评论数量
• 最近活动时间线
🏢 组织管理
组织列表页：我所属的所有组织，显示角色（owner/admin/member）
组织详情页：
  • 基本信息（名称、头像、描述、网站）
  • 成员管理（邀请/移除/改角色）← 仅 owner/admin 可操作
  • 组织下的项目列表
创建组织：申请创建 → 等待运维审核

⚠️ 组织和项目是独立的！组织下可以有项目，但项目不一定属于组织
📦 项目管理
项目列表页：我参与的所有项目（含个人项目 + 组织项目）
项目详情页：
  • 项目信息编辑
  • 项目成员管理（邀请/移除协作者）
  • 项目认领申请（对无主项目）

⚠️ 项目的归属来源清晰标注：个人项目 / 属于 XX 组织
🐛 Bug 反馈
Bug 列表页：我所有项目的 Bug 反馈，可按项目筛选
Bug 详情页：修改状态（open→doing→done）、添加标签、回复
💬 评论管理
评论列表页：我所有项目的用户评论，可按项目筛选
评论操作：删除不当评论、回复评论

📐 组织 vs 项目：关系明确化

🏢 组织
是什么：团队/工作室/公司等集体实体
谁创建：用户申请 → 运维审核
内部层级：owner → admin → member
能做什么：
  • 管理组织信息
  • 管理组织成员
  • 组织下的项目自动继承成员
独立页面：/dev/organizations
📦 项目
是什么：一个具体的软件/工具/资源
谁创建：用户提交 / 运维收录
归属模式：
  • 个人项目 → 直接关联用户
  • 组织项目 → 关联组织
  • 也可以两者都有
能做什么：
  • 编辑项目信息
  • 管理项目成员
  • 处理 Bug 和评论
独立页面：/dev/projects

🔗 组织-项目关联方式

场景 1：个人独立开发
张三提交项目 A → project_members: (project_A, user_张三, owner)
项目 A 不属于任何组织，张三一个人管理
场景 2：组织项目
组织 X 创建项目 B → project_members: (project_B, org_X, owner)
组织 X 的所有成员自动获得项目 B 的管理权限
场景 3：组织项目 + 外部协作者
组织 X 的项目 B + 邀请李四协助 →
project_members: (project_B, org_X, owner) + (project_B, user_李四, collaborator)
李四不是组织成员，但可以管理这个项目
场景 4：多组织开发者
王五属于组织 X 和组织 Y →
organization_members: (org_X, user_王五, member) + (org_Y, user_王五, admin)
王五在开发者后台能看到两个组织的项目

📌 关键原则

✅ 组织和项目在后台是两个独立的顶级导航项，不嵌套
✅ 组织页面只管组织相关的事（信息、成员）
✅ 项目页面只管项目相关的事（信息、成员、Bug、评论）
✅ Bug 和评论也是独立的顶级导航项，跨项目聚合展示
✅ 组织下的项目在"项目管理"页面里用标签标注归属
✅ 权限传递：组织成员 → 自动获得组织下项目的管理权限
