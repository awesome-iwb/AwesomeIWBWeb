三种设计方案详细对比

结合"项目-组织-开发者"关系的深度思考

🤔 核心问题：项目归属模型

一个项目可能属于个人，也可能属于组织/团队。
如果属于组织，组织下有多个开发者，谁能管理这个项目？
在项目页面上如何显示开发者信息？
运维如何指定项目的开发者？
A

方案 A：资源级 Capability（增量扩展）

优点

  • 改动最小，在现有 capability 体系上扩展
  • 向后兼容，现有运维后台不受影响
  • capability 体系保持统一

缺点

  • 需要区分"全局能力"和"资源级能力"
  • 组织概念需要额外处理
  • 权限校验逻辑变复杂
数据模型
capabilities（现有）
├─ user:comment
├─ user:avatar
├─ dev_panel_access
└─ project:update ← 资源级

project_members（新增）
├─ project_id → projects.id
├─ user_id → users.id
└─ role → owner | collaborator

校验逻辑：
全局能力 → 查 user_capabilities
资源级能力 → 查 project_members
B

方案 B：独立角色 + 资源归属（重构角色）

优点

  • 语义清晰：角色和资源归属分离
  • 开发者身份由项目归属隐式获得
  • 天然支持"有项目=开发者，无项目=普通用户"

缺点

  • 需要重构 role 推断逻辑
  • dev 不再是全局角色
  • 运维手动分配开发者权限的场景需要特殊处理
数据模型
users.role → 只保留 user | ops
dev 不再是全局角色

project_members（新增）
├─ project_id → projects.id
├─ user_id → users.id
└─ role → owner | collaborator

校验逻辑：
有项目成员记录 → 自动获得
dev_panel_access + 项目级权限
无项目成员记录 → 普通用户
C

方案 C：混合方案 + 组织支持（最完整）

优点

  • 最完整的模型，支持组织概念
  • capability 做全局权限，project_members 做资源级权限
  • 保留 dev 作为全局角色，同时支持项目级归属
  • 项目可属于个人或组织

缺点

  • 复杂度最高
  • 需要新增 organizations 表
  • 开发量最大
数据模型
users.role → user | dev | ops
保留 dev 全局角色

organizations（新增）
├─ id, name, slug, avatar_url
└─ description

organization_members（新增）
├─ org_id → organizations.id
├─ user_id → users.id
└─ role → owner | admin | member

project_members（新增）
├─ project_id → projects.id
├─ user_id → users.id 或
├─ org_id → organizations.id
└─ role → owner | collaborator

💡 关于"组织"的思考

你提到的"项目可能属于组织"是个很好的点。这引出了两种子模型：

子模型 1：扁平模式 — 项目直接关联开发者，不引入组织概念。一个项目有多个成员，简单直接。

子模型 2：组织模式 — 引入组织/团队概念，项目可属于组织，组织下有多个成员。类似 GitHub 的 Organization 模型。

⚠️ 建议：可以先做扁平模式（MVP），后续再扩展组织模式。组织模式可以渐进式加入，不需要一开始就做。
